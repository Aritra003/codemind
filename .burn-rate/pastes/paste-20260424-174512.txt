# CodeMind: Add `codemind watch` Command

## BUDGET: $0. This feature uses ZERO API calls by default. The watch loop runs the fast tier only (graph traversal, no LLM). The optional `--think-on-critical` flag triggers ONE Opus call only when a CRITICAL change is detected — but don't test that flag during this sprint.

## IMPORTANT: Do NOT modify any existing commands, runners, or core logic. This is a NEW command that IMPORTS existing functions. If you need to change any existing file, limit it to exporting a function that was previously internal.

---

## What to Build

A new `codemind watch` command that runs as a long-lived background agent. It watches for file saves, runs blast radius analysis on the changed file, and prints a real-time risk alert in the terminal. When risk is HIGH or CRITICAL, it shows the full check output. When risk is LOW or MEDIUM, it shows a one-line summary.

This targets the "Best Use of Claude Managed Agents" prize: a long-running, autonomous task that you'd actually ship — not a demo, but something a developer runs in a terminal pane all day.

## Architecture

```
codemind watch
    │
    ├── Loads the existing graph from .codemind/graph.msgpack (once, at startup)
    │
    ├── Uses fs.watch / chokidar to monitor source files for changes
    │   (debounce: 2 seconds after last save, to avoid rapid-fire triggers)
    │
    ├── On file change:
    │   1. Re-parse ONLY the changed file (incremental: update its nodes/edges in the graph)
    │   2. Run the existing check logic (fast tier) on the changed file
    │   3. Display result based on risk level:
    │      - LOW: one-line green summary, no interruption
    │      - MEDIUM: one-line yellow summary with dependent count
    │      - HIGH: full check output (the table with dependents + gaps)
    │      - CRITICAL: full check output + terminal bell + suggestion to run --think
    │
    └── Keeps running until Ctrl+C
```

## Key Implementation Details

### Debouncing
File saves often trigger multiple change events (editor writes temp file, then renames). Use a 2-second debounce per file path. If the same file changes 5 times in 2 seconds, only analyze once.

```typescript
const pending = new Map<string, NodeJS.Timeout>();

watcher.on('change', (filePath: string) => {
  if (pending.has(filePath)) clearTimeout(pending.get(filePath)!);
  pending.set(filePath, setTimeout(() => {
    pending.delete(filePath);
    analyzeChange(filePath);
  }, debounceMs));
});
```

### Incremental Re-parse
Do NOT re-index the entire repo on every file save. Re-parse ONLY the changed file:
1. Remove all nodes and edges FROM the changed file in the graph
2. Re-parse the changed file with tree-sitter
3. Re-add the new nodes and edges
4. Run check on the new nodes

Use existing parser functions. If the parser currently only works on entire repos, extract the single-file parsing into a reusable function that the watcher can call. This is the ONE place you might need to refactor an existing file — extract a `parseFile(filePath)` function from the indexer if one doesn't already exist.

### File Filtering
Only watch files with extensions matching indexed languages (.ts, .tsx, .js, .jsx, .py). Ignore:
- `node_modules/`
- `.git/`
- `.codemind/`
- `dist/`, `build/`, `.next/`, `out/`
- `*.test.*`, `*.spec.*` (optional: include tests if the user passes `--include-tests`)
- `*.json`, `*.md`, `*.yaml`, `*.css`

### Process Lifecycle
- Handle SIGINT (Ctrl+C) gracefully: print summary and exit
- Handle SIGTERM: same
- On exit, print a summary: "Watched for 23 minutes. 14 changes analyzed. 2 HIGH risk alerts."

```typescript
let changeCount = 0;
let highAlerts = 0;
const startTime = Date.now();

process.on('SIGINT', () => {
  const minutes = Math.round((Date.now() - startTime) / 60000);
  console.log(`\n  Watched for ${minutes} minutes. ${changeCount} changes. ${highAlerts} high-risk alerts.`);
  process.exit(0);
});
```

```bash
# Revert changes
cd ~/test-repos/got && git checkout .
```

**Pass criteria:**
- [ ] Watch starts without crashing, shows startup banner
- [ ] Detects file change within 3 seconds of save
- [ ] Shows correct risk level (options.ts should be HIGH, index.ts should be LOW/MEDIUM)
- [ ] Does NOT re-index entire repo on each change (check by timing — should be <1s per change)
- [ ] Ctrl+C prints summary and exits cleanly
- [ ] Does NOT trigger on node_modules, .git, or non-source files
- [ ] Multiple rapid saves to same file only trigger ONE analysis (debounce works)
- [ ] No API calls made (verify: works without ANTHROPIC_API_KEY set)

## MCP Tool Addition

Add ONE new MCP tool for the watch status (do NOT add a tool that starts/stops watching — that's a CLI concern):

```typescript
{
  name: "codemind_watch_status",
  description: "Check if CodeMind watch is running and get its latest alerts",
  input_schema: { type: "object", properties: {} }
}
```

This tool reads from a small status file that the watcher writes:
`.codemind/watch-status.json` — updated on every change analysis:
```json
{
  "running": true,
  "started_at": "2026-04-24T14:30:00Z",
  "changes_analyzed": 14,
  "high_alerts": 2,
  "last_change": {
    "file": "src/auth/middleware.ts",
    "risk": "HIGH",
    "dependents": 38,
    "at": "2026-04-24T14:34:22Z"
  }
}
```

This way Claude Code can ask "is watch running? any high-risk changes recently?" without needing to interact with the watch process directly.
